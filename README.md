# Adaptive Conversion Feed

A research prototype exploring whether an AI system can discover more effective arguments for a specific person from a short sequence of behavioral signals.

**Live demo: [adaptive-conversion-feed.vercel.app](https://adaptive-conversion-feed.vercel.app/)**

The app presents a fictional product, **FocusFlow**, as a scrolling landing page. Each full-screen section is generated by an LLM and uses a different persuasion angle. The visitor can scroll down, scroll back up, close the experiment, or click the single call to action: **Sign up for FocusFlow**.

The project is not a chatbot. It is closer to an adaptive landing page or a vertical feed: the system watches how someone moves through the page, infers what appears to be working, and generates the next section accordingly.

## What it does

Adaptive Conversion Feed is a client-side research prototype that simulates an adaptive landing page for FocusFlow, a fictional productivity product.

A visitor sees a sequence of large, visually polished landing-page sections. Each section contains:

- a headline
- a subheadline
- body copy
- one persuasion angle label
- one CTA button: **Sign up for FocusFlow**

There is no “next” button. Moving through the experience is done by scrolling, which makes the prototype feel more like a real landing page and less like an explicit survey or lab task.

As the visitor scrolls, the system records behavioral signals such as dwell time, scroll depth, direction of travel, revisits, closing, and CTA conversion. When the visitor reaches the end of the generated content, the app prefetches the next LLM-generated section and appends it below the current one.

The experiment can end in two ways:

1. **Conversion:** the visitor clicks **Sign up for FocusFlow**.
2. **Close:** the visitor clicks the X button near the top-right, simulating a tab close. This leads to an “Experiment complete” screen.

In development mode, both end states can expose the research dashboard and exported JSON.

## Why it exists

The core question is:

> Can an AI system adapt persuasive content to an individual visitor using only implicit interaction signals?

Traditional landing pages are static. A/B tests can optimize across populations, but they do not usually adapt within one session. This project explores whether a model can use early behavioral feedback to choose a better next argument for a single person.

The prototype treats persuasion as a sequence problem:

- **State:** the visitor’s interaction history
- **Action:** the next landing-page section
- **Signal:** dwell, scroll depth, revisits, advances, close, or CTA
- **Goal:** understand which content seems to move the visitor closer to conversion

The output is not only a conversion event. The research value comes from the full behavioral trace and the final content-focused interpretation of what appeared to resonate.

## Research design

The project tests adaptive persuasion through a fictional product so that the mechanics can be studied without selling a real thing.

The model rotates through a set of persuasion angles:

- social proof
- convenience
- curiosity
- identity
- aspiration
- cost savings
- fear of missing out
- objection handling
- evidence
- testimonial

The app deliberately relies on behavior rather than self-report. It does not ask visitors to rate interest, explain their thoughts, or move a slider. The idea is to preserve a more natural browsing experience where the visitor simply reads, scrolls, returns to earlier sections, leaves, or signs up.

The prototype does not use eye tracking, webcam input, biometrics, emotion recognition, mouse tracking, or hidden personal data.

## User experience

The experience has four phases:

```text
intro -> feed -> converted
             -> ended
```

### Intro

The visitor sees a one-time disclosure explaining that the page adapts to their behavior. The visitor then starts the experiment.

### Feed

The visitor scrolls through full-screen sections. Each section is designed to look like a modern landing-page block, not a literal card. The only CTA button is **Sign up for FocusFlow**.

The feed supports scrolling both down and up. Scrolling up is important because returning to an earlier section can indicate comparison, hesitation, or renewed interest.

### Converted

If the visitor clicks the CTA, the session enters the converted state. In development mode, the researcher can view metrics after the final profile has finished compiling.

### Ended

If the visitor clicks the X button, the app logs a clean `leave` event and shows an “Experiment complete” screen. In development mode, the researcher can view the same dashboard and export the same JSON.

## Architecture

**Stack:**

- Next.js App Router
- TypeScript
- Tailwind CSS
- shadcn-style UI primitives
- Gemini API through a server-side route
- Client-memory session state
- JSON export for analysis

```text
app/
  page.tsx                 Phase orchestration and top-level UI
  layout.tsx               App metadata and fonts
  globals.css              Global styles
  api/generate/route.ts    Server route for model generation

components/
  IntroScreen.tsx          One-time disclosure screen
  PitchCard.tsx            Full-screen adaptive landing-page section
  LoadingCard.tsx          Loading section while the next card is prefetched
  ConvertedScreen.tsx      Post-CTA end state
  EndScreen.tsx            Post-close end state
  MetricsView.tsx          Shared metrics and final profile view
  ResearchDashboard.tsx    Dev-only floating research panel
  FullJsonBlock.tsx        Copyable full JSON block
  ui/
    button.tsx             Button primitive
    spinner.tsx            Spinner primitive

lib/
  useSession.ts            Session state machine, event log, dwell tracking, export
  metrics.ts               Pure metric derivation from records and events
  gemini.ts                Generation, fallback cards, reflections, final profile
  product.ts               Product metadata and angle styling
  utils.ts                 Shared utility helpers

types/
  index.ts                 Shared TypeScript types
```

The main architectural principle is that the event log is the source of truth. The UI, dashboard, export, timeline, and profile are all derived from the same underlying session data.

## Session lifecycle

`useSession.ts` owns the session state machine.

At a high level, it handles:

- current phase
- generated card records
- event log
- active section tracking
- dwell accumulation
- scroll-depth tracking
- next-section prefetching
- CTA clicks
- close events
- post-card reflections
- final profile generation
- JSON export

The feed uses a single active-section model. Rather than treating every intersection event independently, the app centralizes section transitions through the active card logic. This avoids duplicate events and prevents wrong-card attribution when multiple sections cross thresholds during a fast scroll.

A terminal latch makes conversion and close idempotent. Once a terminal action happens, additional observer events cannot corrupt the final record.

## Metrics

All metrics are derived from the event log and card records in `lib/metrics.ts`.

### Primary metric

- Cards viewed before CTA click

### Conversion and session metrics

- Converted or not converted
- Session length
- Number of sections viewed
- Drop-off section index
- CTA angle
- Final outcome

### Timing metrics

- Average dwell time per section
- Median dwell time per section
- Total dwell time
- Longest dwell
- Shortest dwell before an advance
- Expected read time
- Dwell ratio compared with expected read time

### Advance behavior

An `advance` means the visitor moved downward past a section. It is intentionally not treated as a rejection by default.

A downward advance could mean:

- the section did not land
- the section was understood but not enough to convert
- the visitor wanted more information before deciding
- the visitor was comparing multiple claims

The metrics therefore separate advances into:

- quick advances
- engaged advances
- total advances

The threshold is controlled by `ENGAGED_DWELL_MS` in `metrics.ts`.

### Scroll-up behavior

The app tracks when the visitor returns to earlier content.

Relevant metrics include:

- scroll-back count
- revisited section count
- visit order per section
- arrival direction
- exit direction

Scroll-up behavior is especially useful because revisits can reveal which claims or promises stayed mentally active after the visitor moved on.

### Per-angle summary

For each persuasion angle, the dashboard can summarize:

- number of times used
- average dwell
- average scroll depth
- outcomes associated with the angle

These summaries are descriptive, not causal. One session is not enough to prove that an angle works better in general.

### Timeline

The step-by-step timeline is one of the most useful research outputs. It reconstructs the session as a sequence of visits:

- step number
- card index
- visit number
- angle
- headline
- subheadline
- body
- arrival direction
- exit direction
- dwell for that visit
- outcome
- reflection, when available

Possible outcomes include:

- `advanced`
- `scrolled_back_away`
- `signed_up`
- `closed`
- `still_viewing`

## Adaptive generation

The model generates one landing-page section at a time.

On each prefetch, the client sends a compact adaptation context to the server route. The context includes:

- product metadata
- prior card history
- angles used
- remaining angles
- dwell and scroll signals
- revisit information
- derived session stats
- prior reflections, when available

The server calls the model and expects structured JSON.

Example card output shape:

```json
{
  "analysis": "Reasoning about the session so far.",
  "angle": "evidence",
  "headline": "A clear headline for this section.",
  "subheadline": "A supporting line.",
  "body": "Landing-page body copy.",
  "inferred_user_state": "Short model interpretation.",
  "reasoning_summary": "Why this section was chosen."
}
```

The system enforces angle variety in code. The model receives unused angles as the allowed set, and if it repeats an exhausted angle, the server can retry or fall back to an allowed one.

If there is no API key or the model returns invalid output, the app uses built-in fallback sections. This means the UI, metrics, timeline, dashboard, and export still work without a live model.

## Reflections and final profile

The project has three model-related layers:

1. **Generation:** creates the next landing-page section.
2. **Post-action reflection:** analyzes how the visitor reacted to a completed section.
3. **Final profile:** summarizes what seemed to resonate or fail across the whole session.

The post-action reflection runs after a section receives an outcome such as advance, scroll-back-away, signed-up, or closed. These reflections are research artifacts and can also inform later generations.

The final profile is compiled after conversion or close. It focuses on product substance rather than generic persuasion labels. The goal is not merely to say “the user liked aspiration.” It should explain what content appeared to matter, such as:

- which FocusFlow promise held attention
- what claim may have created interest
- what wording or concept seemed to pull the visitor back
- what did not click
- what might have caused hesitation
- what factor most likely caused the CTA click, if any

Because the final profile is asynchronous, the end screens use a loading state. If the researcher tries to view metrics immediately, the UI waits until the profile is compiled instead of showing a half-finished analysis.

## Research dashboard

In development mode, the app exposes a research dashboard.

The dashboard includes:

- primary conversion metrics
- timing metrics
- advance behavior
- scroll-up and revisit behavior
- angle summaries
- step-by-step timeline
- latest model inference
- reflections
- final profile, when available
- copyable full JSON
- export session JSON button

The dashboard is not meant for normal participants in a production study. In production, it should be hidden or replaced with a proper logging pipeline.

## Ethics and safety stance

This project studies adaptive persuasion, which is powerful and potentially sensitive. The implementation uses several guardrails:

- A one-time disclosure tells visitors the page adapts to their behavior.
- The product is fictional.
- The prompt forbids fabricated statistics, fake named customers, fake scarcity, and exploitation of fear or insecurity.
- The prototype avoids biometrics, webcam signals, emotion recognition, and hidden personal data.
- State stays in the browser unless the researcher exports it.

The point is to study adaptation and preference inference, not to build a covert manipulation system.

If this were used with real research participants, it should be reviewed as a human-subjects study and deployed with explicit consent, data handling rules, and debriefing.

## Setup

```bash
npm install
cp .env.example .env.local
npm run dev
```

Open the local development URL shown by Next.js, usually:

```text
http://localhost:3000
```

Without a Gemini API key, the app still runs using fallback cards.

## Environment variables

```bash
GEMINI_API_KEY=your_key_here
```

The API key is read only on the server side. It is not sent to the client.

## Deployment

A typical deployment flow:

1. Push the project to a Git repository.
2. Import the repository into Vercel or another Next.js-compatible host.
3. Add `GEMINI_API_KEY` as an environment variable.
4. Deploy.

For real data collection, do not rely only on client-side export. Add a secure logging endpoint and send completed session payloads there with appropriate consent and privacy controls.

## Known limitations

- The app stores state in client memory only. Refreshing the page loses the session unless it has been exported.
- Tab-close capture through browser events is best-effort. The in-app X button is the reliable close path.
- Per-angle effectiveness is descriptive within one session. It is not causal evidence.
- The active-section threshold assumes full-screen sections.
- LLM-generated analysis can be useful but should not be treated as ground truth.
- Final profiles are interpretations of behavior, not verified statements about the visitor.
- Model pricing and availability can change. Check the provider’s current pricing before estimating real deployment costs.
- The prototype does not include authentication, persistent storage, participant IDs, or a study-management interface.

## Future directions

- Add a secure logging backend for production studies.
- Add participant consent and debrief flows.
- Compare adaptive sessions against static landing-page controls.
- Run simulated-user sessions with hidden personas.
- Add randomized static baselines for each persuasion angle.
- Analyze whether revisits predict conversion better than dwell alone.
- Generate visual variants as well as copy variants.
- Add a researcher interface for comparing exported sessions.
- Use multi-session data to estimate angle effectiveness across populations.
- Add privacy-preserving participant IDs for longitudinal experiments.
